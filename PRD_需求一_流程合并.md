## 一、背景与目标

### 1.1 现状痛点

业务员需跨两个系统完成一次付款申请：

| 步骤 | 系统 | 操作 |
|------|------|------|
| 1 | BMS | 选择明细记录 → 填写标题/说明 → 申请结算 |
| 2 | OA | 重新填写唯普付款审批表 → 发起审批流 |

财务同样需要在两个系统分别审核，存在重复劳动。  
业务量：每月约 **50个凭证、11100条明细**，人工处理压力较大。

### 1.2 目标

将OA付款审批流程完整迁移至BMS，业务员在BMS内一站式完成：

>明细确认 → 发票上传 → 付款信息录入 → 发起审批

全程不再登录OA系统。

### 1.3 涉及范围

- **涉及模块**：BMS >报销核算中心 >批量结算
- **涉及角色**：业务员、部门负责人、财务人员、公司领导、系统管理员
- **不涉及**：U8凭证生成（见需求二）、OA系统下线或数据迁移

## 三、目标流程（To-Be）

### 3.1 BMS原有明细流程

```
业务员
  └─ 在BMS勾选明细记录（待提交）
       └─ 填写申请表单（标题、说明、发票、付款信息）
            └─ 点击"提交审批"
                 └─ 部门负责人审批
                      ├─ 驳回 → 退回业务员
                      └─ 通过 → 财务人员审批
                               ├─ 驳回 → 退回业务员
                               └─ 通过 → 公司领导审批
                                         ├─ 驳回 → 退回业务员
                                         └─ 通过 → 状态变"待支付"
                                                    └─ 财务导出PDF/打印归档
```

### 3.2 外部对账单导入流程（新增）

```
业务员
  └─ 点击"导入对账明细"
       └─ 选择合作方 + 结算年月 + 上传Excel
            └─ 系统校验 → 预览确认 → 导入入库（状态：待提交）
                 └─ 勾选记录 → 申请结算
                      └─ 进入与上方相同的审批流程
```

## 四、状态流转

```
待提交 ──→ 待部门审批 ──→ 待财务审批 ──→ 待领导审批 ──→ 待支付
                │                │                │               │
              驳回             驳回             驳回       财务打回（FNC侧）
                └──────────────────────────────────────→ 已驳回（退回业务员）
                                                                ├─ 修改后重新提交
                                                                └─ 废弃（终态）
```

| 状态 | 说明 |
|------|------|
| 待提交 | 明细已录入/导入BMS，尚未发起申请 |
| 待部门审批 | 业务员已提交，等待部门负责人 |
| 待财务审批 | 部门已通过，等待财务审核 |
| 待领导审批 | 财务已通过，等待公司领导 |
| 待支付 | 全流程审批通过；财务可通过FNC发起"打回"将状态退回"已驳回"（见5.3.3） |
| 已驳回 | 任一审批节点驳回，或财务从FNC发起打回，退回业务员处理 |
| 已废弃 | 业务员主动废弃，终态，不可恢复 |

## 五、功能需求详述

### 5.1 申请表单改造（新增字段）

**位置**：现有"填写表单"页（勾选明细后进入）

在现有"标题"和"结算说明"字段基础上，新增以下内容：

#### 5.1.1 发票上传

| 项目 | 要求 |
|------|------|
| 文件格式 | 支持 JPG、PNG、PDF |
| 上传数量 | 单次可上传多张，建议上限20张（可配置） |
| 单文件大小 | 不超过10MB |
| 展示方式 | 缩略图列表，支持单张预览、删除 |
| 是否必填 | 必填，未上传不允许提交审批 |

#### 5.1.2 付款信息

参考OA"唯普付款审批表"字段，新增如下（均为必填）：

| 字段 | 类型 | 说明 |
|------|------|------|
| 收款方名称 | 下拉搜索 | **与需求二PRD中"收款单位"为同一字段，统一命名为"收款方名称"**。必须从合作方列表选择（支持关键字搜索），选中后自动带入开户行和银行账号，同时作为凭证摘要中"{收款公司全称}"的来源。如收款方不在列表中，须先由管理员在合作方管理后台（5.5）维护后方可选择，**不支持表单内临时手填**（以确保摘要和账号信息的准确性）。 |
| 收款方开户行 | 文本（自动带入） | 选择收款方名称后自动填入，不可手动修改 |
| 收款方银行账号 | 文本（自动带入） | 选择收款方名称后自动填入，不可手动修改 |
| 付款金额（元） | 数字（**只读**） | **自动计算，等于所选明细记录含税结算金额之和，不允许手动修改。** 若实际付款金额与明细不符，须先在对应明细中调整单车金额，确保"付款金额 = Σ明细含税金额"始终成立，防止后续FNC生成U8凭证时借贷不平（见需求二PRD 4.2）。 |
| 付款种类 | 下拉选择 | 如：对外支付、内部付款（与OA保持一致，后台可配置） |
| 付款项目说明 | 文本 | 填写用途，如"深圳区域好快帮过户款"（原"结算说明"中的用途描述迁移至此字段） |

#### 5.1.3 表单操作按钮

| 按钮 | 行为 |
|------|------|
| 保存草稿 | 保存填写内容，状态保持"待提交"，不发起审批 |
| 继续添加车辆 | 保持现有功能不变 |
| 提交审批 | 校验必填项后发起审批流，状态变为"待部门审批" |

### 5.2 审批流程

#### 5.2.1 审批节点配置

| 审批层级 | 审批人 | 匹配规则 |
|----------|--------|---------|
| 第一级 | 部门负责人 | 按申请人所属部门自动匹配 |
| 第二级 | 财务人员 | 固定角色，后台配置 |
| 第三级 | 公司领导 | 固定角色，后台配置 |

>审批人由系统管理员在后台维护，支持按费用类型或结算机构配置不同审批人。
>每个审批节点均支持 **BMS PC端** 和 **企业微信移动端**（唯普汽车-审核）双端操作，领导审批以移动端为主。

#### 5.2.2 审批人可见信息

每个审批节点，审批人可查看：

- 申请基本信息（标题、申请人、申请时间、结算机构、合计金额）
- 明细列表（车牌号、费用类型、金额等）
- 上传的发票（支持在线预览）
- 付款信息（收款方、账号、金额等）
- 审批历史（前序节点的审批人、时间、意见）

#### 5.2.3 审批操作

| 按钮 | 行为 |
|------|------|
| 通过 | 流转至下一节点；末级通过后状态变"待支付" |
| 驳回 | 弹出驳回原因输入框（必填），提交后退回申请人 |

#### 5.2.4 消息通知与审批渠道

**支持双端审批**：审批人可在 **BMS PC端** 或 **企业微信移动端**（唯普汽车-审核）完成审批操作，两端数据实时同步。

| 触发事件 | 通知对象 | 企业微信推送内容 |
|----------|----------|----------------|
| 业务员提交审批 | 第一级审批人 | "【批量结算付款审批】您有一条结算申请待审批，申请人：XXX，金额：XXX元" |
| 每级审批通过 | 下一级审批人 | "【批量结算付款审批】您有一条结算申请待审批，申请人：XXX，金额：XXX元" |
| 全部审批通过 | 申请人 | "您的结算申请已审批通过，状态：待支付" |
| 审批被驳回 | 申请人 | "您的结算申请被驳回，原因：XXX" |

>通知渠道：企业微信消息推送（主要）+ BMS系统内消息（同步）。

#### 5.2.5 企业微信审批模板（新增）

在"唯普汽车-审核"企业微信审批应用中，新增审批类型：**批量结算付款审批**。

**移动端待审批列表展示**（参考现有样式）：

| 展示字段 | 内容 |
|---------|------|
| 申请人 | 姓名（所属部门） |
| 申请说明 | 付款月份 + 收款单位 |
| 金额 | 含税合计金额，右侧展示 |

**移动端审批详情展示**：

| 区块 | 展示字段 |
|------|---------|
| 基本信息 | 申请人、申请时间、结算机构、费用类型、结算年月、收款方、含税合计 |
| 付款信息 | 收款方名称、开户行、银行账号、付款种类、付款项目说明 |
| 明细摘要 | 明细条数、合计金额（明细列表因数据量大可折叠展示） |
| 审批流节点 | 各节点审批人姓名、角色、操作结果（通过/驳回）、时间（参考现有"审批流程"底部弹层样式） |

**移动端审批操作**：通过 / 驳回（驳回需填写原因）

>模板字段的最终排版由企业微信应用侧配置，以上为内容要求，视觉样式参考现有审批类型。

### 5.3 驳回与废弃处理

#### 5.3.1 被驳回后

申请状态变为"已驳回"，申请人在详情页可查看驳回原因，并选择：

| 操作 | 行为 |
|------|------|
| 修改后重新提交 | 可修改所有字段（明细、发票、付款信息），重新提交后重走完整审批流 |
| 废弃 | 申请终止，状态变"已废弃"，不可恢复；关联明细处理如下：**BMS原有明细**恢复为"待提交"状态；**外部导入明细**同样恢复为"待提交"状态（重新出现在"导入对账明细列表"中等待下次提交结算），**不删除**导入记录。 |

#### 5.3.2 主动撤回

- 审批进行中（待部门/财务/领导审批），申请人可主动撤回并废弃
- 已通过（待支付）的申请不可自行废弃，需由财务通过"财务打回"操作处理（见5.3.3）

#### 5.3.3 财务打回（FNC侧逆向退回）

**触发场景**：BMS申请已通过三级审批（状态为"待支付"），财务在FNC预览凭证时发现信息有误（如收款方名称错误、付款项目说明不准确等），此时申请已无法由业务员自行废弃，需财务发起"打回"操作。

**操作入口**（二选一）：
- FNC凭证管理列表 >对应"待开具"凭证行 > "打回"按钮（需求二PRD 5.6）
- BMS批量结算列表 >对应"待支付"申请行 > "打回"按钮（仅财务角色可见）

**打回流程**：

| 步骤 | 说明 |
|------|------|
| 1 | 财务点击"打回"，弹出原因输入框（必填） |
| 2 | BMS申请状态：待支付 → **已驳回**（驳回来源标记为"财务打回"） |
| 3 | 如FNC已创建"待开具"状态的凭证草稿，同步变为"**已取消**" |
| 4 | 申请关联的所有底层明细恢复为"**待提交**"状态 |
| 5 | 企业微信和BMS站内消息通知申请人："您的结算申请已被财务打回，请修改后重新提交，原因：{打回原因}" |
| 6 | 业务员在"已驳回"状态下修改信息后，可重新发起三级审批 |

> **约束**：已推送U8成功（状态"已开具"）的凭证不支持打回，须在U8侧操作反记账。

### 5.4 导出PDF / 打印

**触发时机**：审批全部通过，状态为"待支付"后可操作

**位置**：详情页右上角，新增"导出PDF"和"打印"按钮（现有"导出"按钮保留）

**导出内容**：

| 区块 | 包含字段 |
|------|---------|
| 基本信息 | 标题、申请人、申请时间、结算机构、结算年月 |
| 付款信息 | 付款种类、金额（数字+大写）、收款方名称、开户行、银行账号、付款项目说明 |
| 审批记录 | 各节点审批人姓名、审批时间、审批意见 |
| 明细列表 | 车牌号、费用类型、实收金额、结算机构等 |

>导出格式参考OA"唯普付款审批表"样式，最终模板由UI设计确认。PDF内容仅含文字和表格（基本信息、付款信息、审批记录、明细列表），不含原始发票文件。

### 5.6 外部对账单批量导入

#### 5.6.1 功能背景

门店合作返利类费用（如车唯拍服务费、集团返利等）不在BMS系统内产生业务记录，由合作方按月提供对账单。目前已收集到的对账单格式有5种，字段名称不一，金额字段名分别为：返利金额、服务费、核算费用等。

为避免适配多种格式带来的开发复杂度，采用**统一标准模板导入**方案，由业务方要求合作门店按系统模板整理数据。

#### 5.6.2 入口位置

批量结算列表页新增"**导入对账明细**"按钮，与现有BMS明细入口分开，独立展示。

#### 5.6.3 标准导入模板字段

系统提供可下载的Excel标准模板，字段如下：

| 字段名 | 是否必填 | 格式要求 | 说明 |
|--------|---------|---------|------|
| 车牌号 | 必填 | 标准车牌格式 | 如：粤A12345 |
| VIN码 | 必填 | 17位字母数字 | 车架号 |
| 门店名称 | 必填 | 文本 | 合作门店名称 |
| 费用项目 | 必填 | 从系统下拉选项中选择 | **新增必填字段**，如：服务费、过户费、整理费；选项由系统管理员在后台配置。此字段用于查重判断（同一台车同一个月可能同时存在服务费和过户费两笔独立记录），并与U8科目映射（见需求二PRD 4.3）联动。 |
| 成交商户 | 选填 | 文本 | 买家/成交方名称 |
| 是否售后 | 选填 | 是/否 | — |
| 是否退车 | 选填 | 是/否 | — |
| 责任方 | 选填 | 文本 | 如：门店/平台 |
| 结算金额（元） | 必填 | 正数，保留2位小数 | 本次应结算金额 |
| 备注 | 选填 | 文本 | 其他说明 |

>字段为暂定版本，待业务方与合作门店确认后可调整，调整前需同步更新系统模板。

#### 5.6.4 导入流程

```
业务员点击"导入对账明细"
    └─ 填写导入信息
         ├─ 合作方：从合作方列表选择 或 手动输入
         └─ 结算年月：月份选择器
              └─ 上传Excel文件（仅支持系统标准模板格式）
                   └─ 系统校验
                        ├─ 校验失败
                        │    └─ 展示错误明细（第X行，X字段，错误原因）
                        │         支持下载错误报告，本次不导入
                        └─ 校验通过
                             └─ 展示预览列表（明细条数、合计金额）
                                  └─ 业务员确认 → 点击"确认导入"
                                                    └─ 记录入库，状态：待提交
```

#### 5.6.5 数据校验规则

| 校验项 | 规则 |
|--------|------|
| 必填项 | 车牌号、VIN码、门店名称、结算金额不能为空 |
| 金额格式 | 必须为正数，最多2位小数 |
| VIN格式 | 17位字母数字组合 |
| 车牌格式 | 符合中国车牌规则 |
| 重复检测 | **同合作方 + 同结算年月 + 同VIN + 同费用项目**四项均相同时，视为重复记录，提示"该记录已存在，是否覆盖"，业务员确认后方可覆盖。（注：同一台车同一个月允许存在不同费用项目的多笔记录，如既有服务费又有过户费，不属于重复数据。） |

#### 5.6.6 导入后记录列表

导入成功的记录进入"**导入对账明细列表**"，与BMS原有业务记录分开展示：

| 字段 | 说明 |
|------|------|
| 合作方名称 | 导入时填写 |
| 结算年月 | 导入时选择 |
| 车牌号 / VIN | 来自导入文件 |
| 门店名称 | 来自导入文件 |
| 结算金额 | 来自导入文件 |
| 状态 | 待提交 / 已提交结算 |
| 导入人 | 操作人 |
| 导入时间 | 系统记录 |

支持按合作方、结算年月、状态筛选查询。

#### 5.6.7 勾选提交结算

业务员核查记录无误后：

1. 勾选需结算的记录（支持全选）
2. 点击"申请结算"
3. 进入与BMS原有明细相同的**填写表单页**（填写标题、说明、上传发票、录入付款信息）
4. 后续走相同的三级审批流程

## 十、验收标准

- [ ] 业务员可在BMS完成发票上传、付款信息录入、提交审批，全程不需登录OA
- [ ] 三级审批流程正常流转，每级审批后正确触发企业微信消息推送和BMS系统内消息
- [ ] 审批人可在企业微信"唯普汽车-审核"中看到"批量结算付款审批"类型，待审批列表展示正确
- [ ] 企业微信移动端审批详情展示完整（基本信息、付款信息、审批流节点），可正常通过/驳回
- [ ] 驳回后业务员可修改重提或废弃，原始明细状态正确恢复
- [ ] 审批通过后可正确导出PDF（包含基本信息、付款信息、审批记录、明细列表），**不含原始发票文件**，内容完整，5秒内生成
- [ ] 详情页"打包下载附件（ZIP）"按钮正常，可将全部发票文件打包下载
- [ ] 批量结算列表状态准确反映当前审批进度
- [ ] 支持上传系统标准模板Excel，导入成功后在独立列表展示
- [ ] 导入数据校验规则生效，错误行可下载错误报告；"费用项目"字段必填校验正常
- [ ] 重复检测规则生效：**同合作方+同结算年月+同VIN+同费用项目**时提示覆盖；同VIN不同费用项目的记录可在同一批次共存，不误报重复
- [ ] 导入记录可勾选发起结算申请，进入正常审批流
- [ ] 合作方管理后台可正常增删改查，业务员填表时可选择并自动带入账号信息；"收款方名称"字段不允许手动填写（必须从列表选择）
- [ ] 废弃后，BMS原有明细和外部导入明细均恢复为"待提交"状态，不删除导入记录
- [ ] 财务人员可对"待支付"申请发起"打回"操作，填写原因后BMS状态退回"已驳回"，底层明细恢复待提交，申请人收到企业微信和站内消息通知
